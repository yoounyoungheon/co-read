# MCP와 Skill

## 같은 자동화 시스템 안에 있지만 역할은 다르다

에이전트 기반 개발 환경을 처음 접하면 `MCP`와 `Skill`이 비슷하게 느껴질 때가 많다. 둘 다 에이전트의 작업 능력을 확장하는 장치처럼 보이기 때문이다. 실제로도 둘은 함께 쓰일 때가 많지만, 맡는 책임은 꽤 다르다.

가장 짧게 말하면 이렇다.

- `MCP`는 에이전트가 호출하는 도구 연결 계층이다.
- `Skill`은 에이전트가 어떤 순서와 기준으로 일을 처리해야 하는지 알려주는 작업 단위다.

즉, MCP가 "무엇을 할 수 있는가"에 더 가깝다면, Skill은 "그 일을 어떤 방식으로 수행해야 하는가"에 더 가깝다. 이 차이를 이해하면 자동화 시스템을 설계할 때 어떤 것을 도구로 만들고, 어떤 것을 스킬로 만들어야 하는지가 훨씬 선명해진다.

이 글에서는 MCP와 Skill의 기초 개념부터 시작해서, 둘의 차이, 함께 동작하는 방식, 실제 예시, 그리고 언제 어느 단위로 설계해야 하는지까지 정리한다.

## 1. MCP는 무엇인가

### MCP는 도구를 표준 방식으로 연결하는 계층이다

MCP는 에이전트가 외부 기능을 사용할 수 있게 해 주는 연결 계층으로 이해하면 가장 쉽다. 에이전트는 기본적으로 텍스트를 읽고 추론할 수 있지만, 실제 개발 작업에서는 그 이상이 필요하다. 파일을 찾고, 문서를 검색하고, 특정 서비스에서 정보를 가져오고, 어떤 시스템의 상태를 읽어야 할 때가 많다.

이때 필요한 것이 MCP다. MCP 서버는 특정 기능을 툴 형태로 노출하고, 에이전트는 그 툴을 호출해 필요한 정보를 가져온다.

예를 들어 이런 것들이 MCP가 되기 쉽다.

- 웹 검색 도구
- 사내 문서 조회 도구
- 피그마 파일 조회 도구
- 레포 내부 가이드 선택 도구
- 배포 상태나 이슈 상태를 읽는 도구

핵심은 MCP가 "행동 규칙"보다는 "기능 제공"에 가깝다는 점이다. 즉, MCP는 어떤 작업 흐름 전체를 정의하기보다, 작업에 필요한 개별 능력을 노출하는 쪽에 더 가깝다.

### MCP는 보통 tool, resource, prompt 같은 인터페이스를 가진다

MCP 서버는 단일 함수 하나만 제공하는 것이 아니라, 보통 몇 가지 다른 방식의 접근 지점을 제공한다.

- `tool`
  - 입력을 받아 어떤 동작을 수행하고 결과를 돌려준다.
- `resource`
  - 문서나 데이터처럼 읽기 중심의 자원을 노출한다.
- `prompt`
  - 특정 작업을 위한 템플릿 프롬프트를 제공한다.

실무에서는 그중에서도 `tool`이 가장 자주 눈에 띈다. 예를 들어 `resolve_guides`, `read_guide`, `web_search_exa` 같은 것은 전형적인 MCP tool이다.

### MCP는 개별 기능 단위로 잘게 나뉘는 편이 좋다

좋은 MCP는 보통 책임이 분명하다. 예를 들어 "프론트엔드 작업 전에 어떤 문서를 읽어야 할지 추천한다"는 기능은 하나의 MCP tool로 잘 정의될 수 있다. 반면 "프론트엔드 작업 전 과정을 전부 알아서 처리한다"는 식으로 덩치가 커지면, 도구라기보다 워크플로우에 가까워진다.

즉 MCP는 대체로 아래 조건에 잘 맞는다.

- 입력과 출력이 비교적 명확하다.
- 하나의 책임을 수행한다.
- 다른 작업 흐름에서도 재사용될 수 있다.
- 사람이 직접 호출해도 이해할 수 있다.

## 2. Skill은 무엇인가

### Skill은 도구가 아니라 작업 방식의 묶음이다

Skill은 보통 "도구 하나"보다 훨씬 큰 단위다. 에이전트가 특정 종류의 일을 할 때 어떤 관점으로 판단해야 하고, 어떤 도구를 먼저 써야 하며, 어떤 실수를 피해야 하는지까지 묶어서 제공하는 작업 템플릿에 가깝다.

예를 들어 `frontend-ui-ux` 같은 skill이 있다고 해 보자. 이 스킬의 핵심 가치는 단순히 CSS를 읽는 함수 하나를 제공하는 데 있지 않다. 오히려 다음 같은 내용을 함께 묶어 주는 데 있다.

- 어떤 UI 변경이 시각 작업인지 판단하는 기준
- 어떤 순서로 레이아웃과 상호작용을 살펴볼지
- 브라우저 확인이나 스크린샷 검증이 필요한지
- 디자인 일관성을 어떻게 판단할지
- 구현 후 어떤 품질 체크를 해야 하는지

즉 Skill은 개별 기능보다도 "작업 운영 방식"을 패키징하는 계층이다.

### Skill은 재사용 가능한 고수준 작업 단위다

Skill은 보통 이런 상황에서 가치가 커진다.

- 같은 종류의 작업을 자주 반복할 때
- 여러 도구를 항상 비슷한 순서로 써야 할 때
- 팀이 원하는 작업 규칙을 표준화하고 싶을 때
- 특정 도메인에서 에이전트가 자주 실수할 때

이 점 때문에 Skill은 일반 개발자가 함수 하나 만들 듯 가볍게 추가하는 단위라기보다, 어느 정도 반복 경험이 쌓인 뒤에 추출되는 경우가 많다. 다시 말해 Skill은 "기능 추가"보다 "숙련된 작업 방식의 캡슐화"에 더 가깝다.

### Skill은 종종 내부적으로 MCP를 사용할 수 있다

Skill과 MCP는 경쟁 관계가 아니다. 오히려 Skill은 내부에서 여러 MCP를 조합해 사용할 수 있다.

예를 들어 피그마 관련 작업을 지원하는 skill이 있다고 하면, 그 skill은 내부적으로 다음을 사용할 수 있다.

- 피그마 노드 정보를 가져오는 MCP
- 디자인 토큰을 읽는 MCP
- 이미지나 프레임을 요약하는 도구
- 코드베이스 안의 기존 컴포넌트를 찾는 검색 도구

즉 Skill은 종종 여러 MCP와 툴을 엮어서 하나의 작업 절차로 만든 상위 계층이다.

## 3. MCP와 Skill의 차이

### 가장 중요한 차이는 책임의 범위다

둘의 차이를 가장 쉽게 구분하는 질문은 이것이다.

> 지금 만들려는 것이 "기능"인가, 아니면 "작업 방식"인가?

이 질문으로 보면 다음처럼 정리할 수 있다.

- `MCP`
  - 기능 제공 중심
  - 명확한 입력/출력
  - 독립 호출 가능
  - 작은 단위로 쪼개기 좋음
- `Skill`
  - 작업 절차와 판단 기준 중심
  - 여러 도구와 규칙을 묶음
  - 도메인 지식과 운영 방식 포함
  - 상대적으로 큰 단위

즉 MCP는 "망치"에 가깝고, Skill은 "목공 작업 절차"에 가깝다. 망치만 있어도 못은 박을 수 있지만, 어떤 순서로 측정하고 자르고 조립할지까지 알려 주지는 않는다.

### 표로 보면 더 명확하다

| 항목        | MCP                            | Skill                                |
| ----------- | ------------------------------ | ------------------------------------ |
| 핵심 역할   | 기능 노출                      | 작업 방식 제공                       |
| 단위        | 비교적 작음                    | 비교적 큼                            |
| 초점        | 무엇을 할 수 있나              | 어떻게 일할 것인가                   |
| 재사용 방식 | 여러 흐름에서 공용 도구로 사용 | 특정 작업군의 표준 절차로 사용       |
| 포함 내용   | tool/resource/prompt           | 절차, 규칙, 체크리스트, 도구 조합    |
| 설계 질문   | 어떤 기능을 API처럼 제공할까   | 어떤 업무를 반복 가능하게 표준화할까 |

## 4. 둘은 실제로 어떻게 함께 동작하나

### 보통은 Skill이 상위에서 흐름을 잡고, MCP가 하위에서 기능을 제공한다

실제 작업에서는 Skill과 MCP가 분리되어 존재하기보다 계층적으로 겹쳐서 동작하는 경우가 많다.

예를 들어 프론트엔드 구현 작업을 생각해 보자.

1. 에이전트는 프론트엔드 작업용 skill을 로드한다.
2. 그 skill은 먼저 전역 규칙을 읽고, 필요한 가이드를 선택하라고 지시한다.
3. 가이드 선택 단계에서 에이전트는 `resolve_guides` 같은 MCP tool을 호출한다.
4. 반환된 guide id를 바탕으로 `read_guide` MCP tool을 호출한다.
5. 그 결과를 근거로 실제 구현과 검증을 수행한다.

이 구조에서 Skill은 "작업 순서"를 결정하고, MCP는 "필요한 정보"를 공급한다.

### 그래서 둘은 대체재가 아니라 계층이 다르다

MCP만 많다고 자동화 시스템이 좋은 것은 아니다. 도구는 많지만, 어떤 상황에 어떤 도구를 어떤 순서로 써야 하는지가 정리되어 있지 않으면 에이전트는 오히려 흔들리기 쉽다.

반대로 Skill만 있고 실제 기능을 제공하는 도구가 부족해도 한계가 있다. 절차는 그럴듯하지만, 실제 파일 검색이나 외부 문서 조회를 할 수 없으면 작업 품질이 떨어진다.

결국 중요한 것은 둘 중 하나를 선택하는 것이 아니라, 어느 책임을 어디에 둘지 분리하는 것이다.

## 5. 구체적인 예시

### 예시 1. 일반 MCP: 웹 검색 도구

`websearch` 같은 MCP를 생각해 보자. 이 도구의 책임은 비교적 단순하다.

- 입력: 검색 쿼리
- 출력: 검색 결과

이건 전형적인 MCP다. 기능이 명확하고, 여러 작업에서 재사용할 수 있으며, 독립 호출이 가능하다.

예를 들어 에이전트는 이런 식으로 사용할 수 있다.

```txt
websearch_web_search_exa(
  query="MCP server design examples for developer tools",
  numResults=5,
  type="auto",
  freshness="any"
)
```

이 호출 자체는 특정 업무 절차를 강제하지 않는다. 그냥 "검색 기능"을 제공할 뿐이다.

### 예시 2. 일반 MCP: 가이드 선택 도구

이 레포에 있는 `frontend-guidance-mcp`도 좋은 예시다. 예를 들어 `resolve_guides`는 프론트엔드 작업 설명과 수정 경로를 받아, 지금 읽어야 할 가이드를 추천한다.

```txt
frontend-guidance-mcp_resolve_guides(
  task="shared/ui button story와 구현을 수정한다",
  targetPaths=[
    "web/src/app/shared/ui/atom/button.tsx",
    "web/src/app/shared/ui/atom/button.stories.tsx"
  ]
)
```

이 도구는 "어떤 guide를 읽어야 하는가"라는 하나의 책임에 집중한다. 이것 역시 MCP다운 설계다.

### 예시 3. Skill: 프론트엔드 UI 작업 스킬

반대로 `frontend-ui-ux` 같은 skill은 단순 도구가 아니다. 이 스킬은 대략 다음을 포함할 수 있다.

- 이 작업이 시각 변경인지 판단하는 기준
- 컴포넌트 구조를 먼저 읽는 순서
- 브라우저 확인이 필요한 상황
- 레이아웃, spacing, typography를 검토하는 기준
- 구현 후 체크해야 할 QA 포인트

즉, 이건 함수 호출 하나로 끝나는 기능이 아니라, 프론트엔드 작업 전체를 안정적으로 수행하기 위한 실행 방식이다.

### 예시 4. 피그마 스킬

질문에서 자주 나오는 것이 "피그마 스킬 같은 것도 skill이냐"는 점인데, 대체로 그렇다. 피그마 연동 자체는 MCP일 수 있지만, "피그마를 읽고 실제 UI 구현으로 연결하는 작업 방식"은 skill에 더 가깝다.

예를 들어 피그마 작업을 지원하는 skill은 이런 흐름을 가질 수 있다.

1. 피그마에서 대상 frame이나 component를 찾는다.
2. spacing, typography, variant 구조를 읽는다.
3. 코드베이스의 기존 컴포넌트와 매칭한다.
4. 이미 있는 디자인 시스템을 우선 재사용한다.
5. 필요한 경우에만 새 컴포넌트를 추가한다.
6. 구현 후 브라우저에서 실제 화면을 검증한다.

여기서 "피그마 파일을 읽는 기능" 자체는 MCP가 될 수 있지만, 위 전체 절차는 skill의 책임이다.

## 6. skill_mcp는 무엇인가

### skill_mcp는 스킬 내부 MCP를 호출하는 브리지다

조금 헷갈리는 지점이 `skill_mcp`다. 이름만 보면 이것도 하나의 일반 MCP처럼 보일 수 있지만, 성격은 조금 다르다. `skill_mcp`는 특정 skill이 내장하고 있는 MCP 서버의 tool, resource, prompt를 호출하기 위한 중계 통로에 가깝다.

즉 차이는 다음과 같다.

- 일반 MCP
  - 세션에 직접 등록된 도구
  - 바로 호출 가능
- `skill_mcp`
  - skill 내부에 붙어 있는 MCP에 접근하는 브리지
  - 먼저 해당 skill이 존재하고, 그 skill이 MCP를 내장해야 의미가 있음

그래서 `skill_mcp`는 그 자체로 업무 기능을 설명하기보다, "스킬 안에 들어 있는 도구에 접속하는 방법"에 가깝다.

### 왜 이런 구조가 필요할까

모든 도구를 세션 전역에 바로 노출하면 편해 보이지만, 실제로는 문제가 생길 수 있다.

- 특정 도메인에서만 쓰는 도구까지 항상 전역으로 보이게 된다.
- 어떤 도구가 어떤 스킬 맥락에서 안전하게 쓰여야 하는지 구분이 흐려진다.
- 스킬 단위로 묶인 워크플로우와 내부 자원을 분리 관리하기 어려워진다.

그래서 어떤 시스템은 공용 도구는 일반 MCP로 제공하고, 특정 스킬에 강하게 결합된 내부 도구는 skill 전용 MCP로 숨겨 두기도 한다.

## 7. 언제 MCP를 만들고, 언제 Skill을 만들어야 하나

### MCP로 만드는 것이 좋은 경우

다음 조건에 가까우면 MCP가 더 적합하다.

- 기능 책임이 하나로 분명하다.
- 입력과 출력이 명확하다.
- 여러 작업 문맥에서 공통으로 재사용된다.
- 사람이 직접 호출해도 용도가 이해된다.

예를 들어 다음은 MCP 후보에 가깝다.

- "현재 수정 경로에 맞는 가이드를 추천한다"
- "피그마 노드 정보를 읽는다"
- "릴리스 노트를 검색한다"
- "사내 위키 문서를 조회한다"

### Skill로 만드는 것이 좋은 경우

다음 조건에 가까우면 Skill이 더 적합하다.

- 여러 도구를 항상 비슷한 절차로 조합한다.
- 도메인별 체크리스트와 운영 규칙이 중요하다.
- 에이전트가 같은 종류의 실수를 반복한다.
- 단순 기능보다 품질 기준과 작업 순서가 더 중요하다.

예를 들어 다음은 Skill 후보에 가깝다.

- "디자인 시안을 읽고 프론트엔드 UI를 구현한다"
- "브라우저에서 재현하고 디버깅한 뒤 최소 수정으로 고친다"
- "PR을 만들기 전에 변경 사항을 종합 검토한다"

### 경험적으로는 작은 기능은 MCP, 숙련 절차는 Skill이 된다

실무적으로 보면 처음에는 MCP가 먼저 생기고, 나중에 Skill이 생기는 경우가 많다. 왜냐하면 처음에는 개별 기능 수요가 먼저 보이기 때문이다. 하지만 팀이 같은 작업을 반복하다 보면, 어느 순간 "도구가 부족한 것"보다 "작업 방식이 매번 흔들리는 것"이 더 큰 문제라는 사실이 드러난다. 그 시점부터 Skill의 가치가 커진다.

## 8. 일반 개발자가 직접 Skill을 만들기 어려운 이유

### Skill은 코드보다 운영 경험이 더 많이 들어간다

일반 개발자가 로컬 유틸 함수를 하나 만드는 것은 비교적 쉽다. 하지만 Skill은 단순 코드 조각이 아니라, 반복적으로 검증된 작업 방식을 문서화하고 자동화하는 단위다. 그래서 단순 구현 역량만으로는 부족할 수 있다.

Skill을 잘 만들려면 보통 이런 것이 필요하다.

- 어떤 단계에서 에이전트가 자주 실패하는지에 대한 관찰
- 실패를 줄이기 위한 작업 순서 정리
- 어떤 도구를 언제 써야 하는지에 대한 기준
- 잘못된 사용을 막는 제한 규칙
- 결과 품질을 판정하는 검증 절차

즉 Skill은 라이브러리처럼 재사용되지만, 순수 라이브러리보다는 "운영 노하우가 담긴 실행 패키지"에 더 가깝다.

### 그렇다고 너무 거창하게만 볼 필요는 없다

다만 Skill이 반드시 거대한 플랫폼 팀만 만들 수 있는 것은 아니다. 작은 팀이라도 특정 업무를 반복하고 있다면, 그 반복되는 절차를 정리해 가벼운 Skill로 발전시킬 수 있다.

예를 들어 다음처럼 시작할 수 있다.

- 먼저 자주 쓰는 검색/조회 기능을 MCP로 분리한다.
- 그 다음 반복되는 작업 순서를 문서화한다.
- 어떤 체크를 빠뜨리면 안 되는지 규칙을 붙인다.
- 마지막으로 그 묶음을 하나의 Skill로 승격한다.

즉 Skill은 처음부터 거대한 제품으로 만들어야 하는 것이 아니라, 반복 작업이 누적되면서 자연스럽게 추출되는 상위 개념으로 보는 편이 현실적이다.

## 9. 설계할 때 자주 하는 오해

### 오해 1. MCP가 많으면 자동화가 좋아진다

아니다. MCP가 많아도 어떤 상황에 어떤 도구를 써야 하는지 정리되어 있지 않으면 오히려 혼란만 커질 수 있다. 도구 개수보다 중요한 것은 책임 분리와 호출 맥락이다.

### 오해 2. Skill은 그냥 큰 라이브러리다

완전히 같지는 않다. 라이브러리는 주로 기능 재사용을 목표로 하지만, Skill은 기능뿐 아니라 절차, 판단 기준, 금지 사항, 검증 방식까지 포함한다. 그래서 Skill은 코드 패키지라기보다 작업 패키지에 가깝다.

### 오해 3. 둘 중 하나만 있으면 된다

실제로는 둘이 함께 있을 때 가장 안정적이다. MCP만 있으면 도구는 있지만 작업 방식이 약하고, Skill만 있으면 절차는 있지만 실제 기능 연결이 약해진다.

## 마무리

MCP와 Skill은 이름이 비슷해서 헷갈리기 쉽지만, 설계 관점에서 보면 역할이 꽤 다르다.

- MCP는 에이전트가 사용할 기능을 표준 방식으로 노출하는 계층이다.
- Skill은 특정 종류의 업무를 안정적으로 수행하기 위한 작업 절차와 규칙의 묶음이다.
- Skill은 종종 여러 MCP를 내부에서 조합해 사용한다.
- 작은 공용 기능은 MCP로, 반복되는 고수준 작업 방식은 Skill로 두는 것이 보통 더 자연스럽다.

결국 중요한 것은 용어 자체보다 책임을 어디에 둘 것인가다. 기능을 도구로 분리할지, 작업 방식을 스킬로 캡슐화할지 명확하게 나누면 자동화 시스템은 훨씬 이해하기 쉬워지고, 유지보수도 쉬워진다.
