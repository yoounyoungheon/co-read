# Frontend Agent Pipeline

## 정책은 짧게, 구현 지식은 선택적으로

프론트엔드 에이전트를 오래 운영하다 보면 같은 문제가 반복된다. 처음에는 `AGENTS.md`에 규칙을 많이 넣을수록 에이전트가 더 잘 동작할 것처럼 보이지만, 시간이 지나면 오히려 반대가 된다. 전역 정책, Storybook 작성법, 디자인 시스템 메타 구조, SSE lifecycle, 스트리밍 payload 포맷, 스타일 디버깅 기준이 한 문서에 계속 쌓이면서 에이전트는 "항상 읽어야 하는 것"과 "이번 작업에서만 필요한 것"을 구분하지 못하게 된다.

이번에 만든 구조는 이 문제를 해결하기 위해 출발했다. 핵심 아이디어는 단순하다.

- 전역적으로 항상 지켜야 하는 판단 규칙은 `web/AGENTS.md`에만 남긴다.
- 구현 방법과 세부 지식은 `frontend-guidance-mcp`에서 가이드로 제공한다.
- 에이전트는 모든 가이드를 읽지 않고, 현재 작업에 필요한 것만 선택적으로 읽는다.
- 가이드는 설명 문서만 바라보지 않고 실제 코드 경로를 근거로 삼고, 필요한 일부 경우에는 생성 산출물도 함께 참조한다.

이 글에서는 이 구조를 왜 설계했는지, 실제 아키텍처는 어떻게 나뉘는지, 에이전트가 어떤 순서로 동작하는지, 그리고 아직 남아 있는 한계는 무엇인지 정리한다.

## 1. 시스템을 설계하게 된 배경

### AGENTS.md 하나로 모든 걸 해결하려고 할 때 생기는 문제

기존 방식의 가장 큰 문제는 "지식의 밀도"보다 "지식의 범위"였다. 프론트엔드 에이전트가 매번 읽는 문서 안에 정책과 구현 상세가 모두 들어 있으면, 에이전트는 매 작업마다 너무 많은 맥락을 한꺼번에 소비하게 된다.

예를 들어 단순히 `shared/ui` 아래 버튼 story를 수정하는 작업을 한다고 해도, 다음 정보가 모두 한 문서에 섞여 있으면 문제가 생긴다.

- Server Component와 Client Component를 어떻게 나눠야 하는지
- 공통 컴포넌트와 도메인 컴포넌트를 어디에 둬야 하는지
- Storybook 메타를 어떻게 써야 하는지
- "디자인 시스템 문서"(`storybook`)를 어떻게 해석해야 하는지
- 공통모듈을 어떻게 사용해야 하는지
- css 속성은 어떻게 적용해야 할지

이 정보들 중 일부는 항상 필요하지만, 대부분은 특정 작업에서만 필요하다. 그런데 문서가 하나면 에이전트 입장에서는 이 구분이 사라진다. 결국 다음과 같은 문제가 생긴다.

- 작업과 무관한 지식까지 항상 읽게 된다.
- 중요한 전역 규칙이 세부 구현 설명에 묻힌다.
- 문서가 길어질수록 유지보수가 어려워진다.
- 구현이 바뀌어도 문서가 뒤늦게 갱신되면 오래된 지식을 그대로 참고하게 된다.

### 특히 프론트엔드에서는 "작업별 맥락 선택"이 중요하다

백엔드 로직은 비교적 좁은 문제 공간 안에서 동작하는 경우가 많지만, 프론트엔드 자동화는 작업 종류가 훨씬 다양하다. 공통 컴포넌트 작업, feature UI 작업, Storybook 작업, 스트리밍 UI 작업, 스타일 디버깅 작업은 서로 요구하는 지식이 다르다.

그래서 이 시스템에서는 "에이전트가 얼마나 많이 아는가"보다 "지금 작업에 맞는 지식을 얼마나 정확하게 골라 읽는가"를 더 중요하게 봤다. 즉, 프론트엔드 자동화의 문제를 지식량의 문제가 아니라 지식 라우팅의 문제로 재정의한 셈이다.

### 설명 문서만 근거로 삼는 구조의 한계

또 하나의 문제는 문서 드리프트였다. 예를 들어 SSE 관련 설명 문서가 과거 이벤트 구조를 기준으로 적혀 있고, 실제 route 구현은 이미 다른 payload 포맷으로 바뀌어 있을 수 있다. 이런 상황에서 에이전트가 문서만 믿고 구현하면, "문서 기준으로는 맞지만 실제 코드 기준으로는 틀린" 결과가 나온다.

그래서 가이드는 단순한 설명 텍스트가 아니라, 실제 코드 경로와 생성 산출물에 연결된 레지스트리여야 했다. 설명은 바뀔 수 있지만, 근거 경로를 같이 들고 있으면 에이전트가 구현 시점에 다시 확인할 수 있기 때문이다.

## 2. 아키텍처 구조

이 구조는 세 층으로 나뉜다.

### 2-1. 정책 계층: `web/AGENTS.md`

`web/AGENTS.md`는 짧은 전역 정책만 담는다. "항상 지켜야 하는 것"만 남겨두었다.

예를 들면 다음과 같은 규칙이다.

- 재사용 컴포넌트는 `src/app/shared/ui`에 둬야 한다.
- 도메인 컴포넌트는 `src/app/feature/*/ui`에 둬야 한다.
- 컴포넌트를 생성하거나 수정하면 같은 경로에 story를 유지해야 한다.
- 기본은 Server Component이고, 상호작용이 필요한 최소 범위만 Client Component로 분리한다.
- 모바일 우선, 가독성 우선으로 구현한다.

중요한 점은 이 문서가 더 이상 Storybook 세부 작성법이나 SSE lifecycle을 설명하지 않는다는 것이다. 이 문서는 "정책 문서"로 역할을 고정한다.

### 2-2. 구현 지식 계층: `frontend-guidance-mcp`

구현 지식은 `mcp` 패키지에 있는 `frontend-guidance-mcp`가 담당한다. 이 서버는 Node.js와 MCP SDK 기반의 stdio 서버로 동작하며, 현재 두 개의 툴만 제공한다.

#### `resolve_guides`

입력:

- `task`: 사용자의 작업 설명
- `targetPaths`: 현재 수정하려는 경로 목록

출력:

- 읽어야 할 guide id 목록
- 선택 이유
- 우선순위

이 툴은 "이번 작업에서 어떤 지식을 우선 읽어야 하는가"를 휴리스틱하게 추천한다.

#### `read_guide`

입력:

- `id`: 읽을 guide id

출력:

- guide 본문
- source path 목록
- canonical source path 목록
- source 존재 여부
- 마지막 갱신 기준

이 툴은 "선택한 가이드를 실제로 읽는 단계"다.

### 2-3. 근거 레지스트리 계층: `mcp/src/guides.ts`

실제 핵심은 `mcp/src/guides.ts`에 있다. 이 파일은 단순 문자열 모음이 아니라, 각 가이드를 다음 구조로 관리한다.

- `id`: 가이드 식별자
- `title`, `summary`, `body`: 사람이 읽는 설명
- `triggers.paths`: 경로 기반 선택 조건
- `triggers.keywords`: 키워드 기반 선택 조건
- `sources`: 참고할 전체 근거 경로
- `canonicalSources`: 실제 최신성 판단에 사용할 핵심 근거 경로

현재 등록된 가이드는 다음 네 가지다.

- `reusable-components`
- `domain-components`
- `sse-chat-streaming`
- `storybook-authoring`

여기서 중요한 설계 포인트는 `sources`와 `canonicalSources`를 분리한 것이다.

- `sources`는 설명 문서, 정책 문서, 실제 코드, 생성 산출물을 모두 포함할 수 있다.
- `canonicalSources`는 실제 구현 판단과 최신성 판단의 기준이 되는 경로만 담는다.

이 분리를 통해 설명 문서가 최신 수정 시각을 가장 최근에 갖더라도, 시스템은 실제 canonical source 기준으로 freshness를 계산할 수 있다. 다만 canonical source가 현재 워크스페이스에 존재하지 않으면 `lastUpdated`는 `null`이 될 수 있다.

### 2-4. 클라이언트 연결 계층

이 MCP 서버는 로컬 클라이언트에서 stdio 기반으로 연결한다. 실제로는 Claude Desktop 설정 파일에 아래와 같은 엔트리를 등록하는 방식이다.

```json
{
  "mcpServers": {
    "frontend-guidance-mcp": {
      "command": "npm",
      "args": ["run", "start"],
      "cwd": "/Users/yun-yeongheon/dev/co-read/mcp"
    }
  }
}
```

즉, 런타임 구조를 요약하면 다음과 같다.

```text
User Request
  -> Agent reads web/AGENTS.md
  -> Agent calls frontend-guidance-mcp.resolve_guides(task, targetPaths)
  -> Agent calls frontend-guidance-mcp.read_guide(id)
  -> Agent uses the returned source paths as implementation evidence
  -> Agent validates with repo-specific checks
```

## 3. 프로세스

이 구조에서 중요한 것은 "문서를 어디에 두었는가"보다 "에이전트가 어떤 순서로 동작하는가"다.

### Step 1. 전역 정책을 먼저 읽는다

에이전트는 항상 `web/AGENTS.md`를 먼저 읽는다. 이 단계의 목적은 구현 상세를 얻는 것이 아니라, 이번 작업에서 반드시 지켜야 할 전역 제약을 먼저 확정하는 것이다.

예를 들어 이 단계에서 에이전트는 다음을 먼저 안다.

- 이 작업이 `shared/ui`인지 `feature/*/ui`인지
- story를 반드시 같이 유지해야 하는지
- Client Component 분리를 얼마나 좁게 해야 하는지
- 모바일 우선, 가독성 우선 원칙을 지켜야 하는지

### Step 2. 작업 설명과 경로로 필요한 가이드를 찾는다

그 다음 `resolve_guides`를 호출한다. 이 단계는 일종의 라우팅 단계다.

예를 들어 다음 입력이 들어오면:

```json
{
  "task": "shared/ui button story와 구현을 수정한다",
  "targetPaths": [
    "web/src/app/shared/ui/atom/button.tsx",
    "web/src/app/shared/ui/atom/button.stories.tsx"
  ]
}
```

서버는 `reusable-components`와 `storybook-authoring`를 우선순위와 함께 반환하도록 설계되어 있다.

현재 선택 기준은 단순한 휴리스틱이다.

- 경로가 어떤 패턴과 매칭되는가
- 작업 설명에 어떤 키워드가 들어 있는가
- story 파일 여부에 대한 보너스 신호가 있는가

필요하면 절대 경로도 받을 수 있고, 서버 내부에서 레포 상대 경로로 정규화한 뒤 매칭한다.

### Step 3. 필요한 가이드만 읽는다

`resolve_guides` 결과를 받은 뒤 에이전트는 반환된 id만 `read_guide`로 읽는 흐름을 따르도록 문서화되어 있다.

이 흐름의 장점은 명확하다.

- 불필요한 지식 로딩을 막을 수 있다.
- 작업별로 필요한 구현 디테일만 읽게 된다.
- 전역 정책과 구현 지식을 분리한 상태를 실제 실행 순서에서도 유지할 수 있다.

### Step 4. 구현은 guide 본문이 아니라 source path를 근거로 진행한다

`read_guide`는 단순히 본문만 주지 않는다. 관련 source path와 canonical source path, 파일 존재 여부, 최신성 기준까지 함께 준다. 다만 이 정보는 선택적 읽기를 강제하는 접근 제어가 아니라, 에이전트가 구현 시점에 근거를 다시 확인할 수 있게 돕는 메타데이터다.

이 구조 덕분에 에이전트는 guide 텍스트를 "최종 정답"처럼 쓰기보다, 구현 시점에 실제 근거 파일을 다시 읽는 흐름을 따르기 쉬워진다. 예를 들어 SSE 가이드는 `docs/stream.md` 같은 설명 문서도 source에 포함하지만, canonical 기준은 실제 route와 shared SSE 코드에 두고 있다.

즉, 가이드는 설명 레이어이면서 동시에 탐색 출발점 역할을 한다.

### Step 5. 구현 후 다시 코드 기준으로 검증한다

마지막 단계는 구현된 레포의 검증 체계에 맞춰 확인하는 것이다. 현재 문서와 MCP 서버는 다음 검증 흐름을 의도하고 있다.

- MCP 서버를 빌드하고 실행 가능한 상태로 유지하기
- stdio 기반 MCP 클라이언트에서 툴을 호출할 수 있게 하기
- `resolve_guides` / `read_guide` 결과를 바탕으로 관련 근거 파일을 다시 확인하기
- 작업 종류에 맞는 코드/빌드 검증을 수행하기

즉, 이 구조는 단순한 문서 정리가 아니라, 실행 가능한 guidance pipeline을 지향하는 구조다.

## 4. 한계 / 개선점

구조는 분명해졌지만, 아직 완성형은 아니다. 운영하면서 개선해야 할 지점도 분명히 있다.

### 4-1. guide registry는 아직 수작업이다

지금은 `mcp/src/guides.ts`에서 guide를 직접 정의한다. 이 방식은 시작하기에는 좋지만, 가이드 수가 늘어나면 유지보수 비용이 커진다.

특히 다음 문제가 생길 수 있다.

- source path를 사람이 직접 관리해야 한다.
- guide 본문과 실제 코드의 관계를 수작업으로 유지해야 한다.
- 새로운 도메인이 추가될 때 trigger와 source를 계속 손으로 붙여야 한다.

개선 방향은 명확하다. Storybook index, metadata 파일, 특정 디렉터리 규칙을 바탕으로 일부 registry를 자동 생성하는 방식으로 가야 한다.

### 4-2. guide 선택 로직은 아직 규칙 기반이다

현재 `resolve_guides`는 경로 패턴과 키워드 중심으로 동작한다. 이 방식은 설명 가능성이 높고 디버깅이 쉽다는 장점이 있지만, 복합 작업에서는 한계가 있다.

예를 들어 하나의 작업이 다음을 동시에 포함할 수 있다.

- shared UI 수정
- feature UI 수정
- story 수정
- 스타일 디버깅

이런 경우에는 단순 score 기반 규칙보다 더 풍부한 작업 분류 계층이 필요할 수 있다. 앞으로는 작업 유형 분류기나 더 세밀한 우선순위 모델을 붙일 여지가 있다.

### 4-3. 설명 문서와 실제 구현의 드리프트는 완전히 사라지지 않는다

이번 구조는 드리프트를 줄였지만, 드리프트를 원천적으로 제거하지는 못한다. 설명 문서도 여전히 source 목록에 남아 있고, 사람은 여전히 설명 본문을 작성한다.

다만 중요한 차이는 있다. 이제 드리프트가 생기더라도 에이전트가 그것을 최종 근거로 삼지 않는다. `canonicalSources`라는 안전장치를 둔 것이 이번 구조의 핵심 개선점이다.

앞으로는 다음을 더 강화할 수 있다.

- canonical source가 바뀌면 guide freshness 경고를 자동으로 띄우기
- source 파일이 없어지거나 경로가 깨졌을 때 CI에서 탐지하기
- guide body와 canonical source 간 차이를 점검하는 lint 추가하기

### 4-4. 지금 구조는 "선택적 읽기"까지는 해결했지만 "선택적 검증"은 더 발전시킬 수 있다

현재는 어떤 guide를 읽을지 잘 고르게 되었지만, 검증 또한 가이드 종류에 따라 더 세밀하게 라우팅할 수 있다.

예를 들어:

- `storybook-authoring`를 읽었으면 story build 또는 story index 확인
- `sse-chat-streaming`를 읽었으면 route payload와 consumer parse 흐름 확인
- `reusable-components`를 읽었으면 shared story와 public prop surface 확인

즉, 다음 단계는 guide selection뿐 아니라 validation selection까지 파이프라인에 포함시키는 것이다.

### 4-5. MCP 의존성이 생긴다는 운영 비용이 있다

이 구조는 분명 더 정교하지만, 단순한 문서 하나를 읽는 방식보다 운영 복잡도는 높다.

- MCP 서버를 실행해야 한다.
- 클라이언트 설정이 필요하다.
- 가이드 레지스트리를 관리해야 한다.
- 로컬과 CI 환경에서 같은 흐름을 재현할 방법이 필요하다.

즉, 이 구조는 "문서를 줄였다"기보다 "문서를 실행 가능한 시스템으로 바꿨다"에 가깝다. 그래서 유지보수 비용도 함께 따라온다.

## 마무리

이번 구조에서 가장 중요했던 변화는 지식을 줄인 것이 아니라, 지식의 책임을 분리한 것이다.

- `web/AGENTS.md`는 항상 지켜야 하는 정책만 담당한다.
- `frontend-guidance-mcp`는 작업 맥락에 맞는 구현 지식을 선택적으로 제공한다.
- `guides.ts`는 각 가이드를 실제 코드와 생성 산출물에 연결한다.
- 에이전트는 이제 모든 걸 한 번에 읽지 않고, 필요한 것만 읽고, 실제 코드를 근거로 구현한다.

프론트엔드 자동화에서 중요한 것은 더 많은 규칙을 한 문서에 넣는 것이 아니라, 에이전트가 지금 필요한 규칙과 지식을 정확히 분리해서 읽게 만드는 것이다. 이 구조는 그 문제를 해결하기 위한 첫 번째 파이프라인이다.
